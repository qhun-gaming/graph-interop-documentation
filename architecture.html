Architecture

GraphInterop follows a layered architecture that separates runtime logic from editor visualization, enabling multiple visual backends.

Design Principles

1. Runtime/Editor Separation

The runtime layer has no dependencies on editor APIs, allowing graphs to execute in builds:

  • Runtime Assembly: QhunGaming.GraphInterop - contains all execution logic

  • Editor Assembly: QhunGaming.GraphInterop.Editor - contains visualization

2. Adapter Pattern

Editor backends implement adapters that translate between the runtime data model and the visual framework:

3. Attribute-Based Configuration

Nodes declare their ports using attributes rather than code:

public class MyNode : GraphNode { [Input(Required = true, ShowEmbedded = true)] public float value; [Output] public float result; }

Thread Safety

  • GraphExecutor: Not thread-safe, execute on main thread only

  • GraphAsset: Not thread-safe during modification

  • TypeCompatibility: Static methods are thread-safe

Memory Considerations

  • Nodes use [SerializeReference] for polymorphic serialization

  • Port definitions are cached per node instance (lazy initialization)

  • Connections are stored by node ID references (no object references)
