GraphInterop Innovations

This page highlights features unique to GraphInterop that go beyond both GraphView and GraphToolkit.

Overview

While GraphInterop adapts GraphToolkit's API and uses GraphView's UI, it also introduces several innovations to improve usability, performance, and developer experience.

Portal System

Problem

In complex graphs, long-distance connections create visual clutter:

This makes graphs hard to read and navigate.

Solution: Send/Receive Portals

GraphInterop introduces portal nodes for invisible connections:

Benefits:

  • Clean visual layout

  • Reduced edge crossings

  • Easier to follow data flow

  • Organizes complex graphs

Implementation

Send Portal:

[NodeInfo( DisplayName = "Send Portal", Category = "Flow", Icon = "d_winbtn_mac_max" )] public class SendPortalNode : GraphNode { protected override void OnDefineOptions(IOptionDefinitionContext context) { context.AddOption<string>("portalId") .WithDisplayName("Portal ID") .WithDefaultValue("Portal") .Build(); } protected override void OnDefinePorts(IPortDefinitionContext context) { context.AddInputPort<object>("value") .WithDisplayName("Value") .Build(); } public override void Process(GraphContext context) { var value = this.GetInputValue<object>("value", context); // Value stored in context, retrieved by ReceivePortal } }

Receive Portal:

[NodeInfo( DisplayName = "Receive Portal", Category = "Flow", Icon = "d_winbtn_mac_min" )] public class ReceivePortalNode : GraphNode { protected override void OnDefineOptions(IOptionDefinitionContext context) { context.AddOption<string>("portalId") .WithDisplayName("Portal ID") .WithDefaultValue("Portal") .Build(); } protected override void OnDefinePorts(IPortDefinitionContext context) { context.AddOutputPort<object>("value") .WithDisplayName("Value") .Build(); } public override void Process(GraphContext context) { // Retrieves value from matching SendPortal var portalId = this.GetOptionValue<string>("portalId"); // Value resolved from context } }

Usage Example

// Graph setup var sendPortal = graph.AddNode<SendPortalNode>(); sendPortal.SetOptionValue("portalId", "PlayerHealth"); var receivePortal = graph.AddNode<ReceivePortalNode>(); receivePortal.SetOptionValue("portalId", "PlayerHealth"); // Connection graph.Connect(healthNode, "health", sendPortal, "value"); // ReceivePortal automatically receives value during execution // Execution dependency // GraphExecutor ensures SendPortal runs before ReceivePortal

Portal Dependency Resolution

GraphExecutor handles portal dependencies in topological sort:

// Pseudo-code for dependency resolution foreach (var receivePortal in graph.GetNodes<ReceivePortalNode>()) { var portalId = receivePortal.GetOptionValue<string>("portalId"); var sendPortal = graph.FindNode<SendPortalNode>(n => n.GetOptionValue<string>("portalId") == portalId ); if (sendPortal != null) { // Add implicit dependency: receivePortal depends on sendPortal AddDependency(receivePortal, sendPortal); } }

This ensures correct execution order without visual connections.

Burst Node Support

Problem

Unity's Burst compiler provides massive performance gains for data-oriented code, but GraphView and GraphToolkit don't support it.

Solution: IBurstNode Interface

GraphInterop adds Burst support via IBurstNode:

public interface IBurstNode { JobHandle ScheduleBurstJob(GraphContext context); void CompleteBurstJob(GraphContext context); }

Implementation Example

using Unity.Burst; using Unity.Collections; using Unity.Jobs; [NodeInfo( DisplayName = "Burst Vector Add", Category = "Math/Burst" )] public class BurstAddVectorsNode : GraphNode, IBurstNode { private NativeArray<Vector3> _input1, _input2, _output; protected override void OnDefinePorts(IPortDefinitionContext context) { context.AddInputPort<Vector3>("a").WithDisplayName("A").Build(); context.AddInputPort<Vector3>("b").WithDisplayName("B").Build(); context.AddOutputPort<Vector3>("result").WithDisplayName("Result").Build(); } public JobHandle ScheduleBurstJob(GraphContext context) { // Allocate native arrays _input1 = new NativeArray<Vector3>(1, Allocator.TempJob); _input2 = new NativeArray<Vector3>(1, Allocator.TempJob); _output = new NativeArray<Vector3>(1, Allocator.TempJob); // Get input values _input1[0] = this.GetInputValue<Vector3>("a", context); _input2[0] = this.GetInputValue<Vector3>("b", context); // Schedule job var job = new AddVectorsJob { Input1 = _input1, Input2 = _input2, Output = _output }; return job.Schedule(); } public void CompleteBurstJob(GraphContext context) { // Set output value this.SetOutputValue("result", _output[0], context); // Dispose arrays _input1.Dispose(); _input2.Dispose(); _output.Dispose(); } [BurstCompile] private struct AddVectorsJob : IJob { [ReadOnly] public NativeArray<Vector3> Input1; [ReadOnly] public NativeArray<Vector3> Input2; public NativeArray<Vector3> Output; public void Execute() { Output[0] = Input1[0] + Input2[0]; } } }

Execution Flow

// GraphExecutor handles Burst nodes specially if (node is IBurstNode burstNode) { var jobHandle = burstNode.ScheduleBurstJob(context); jobHandle.Complete(); // Wait for completion burstNode.CompleteBurstJob(context); } else if (node is IAsyncNode asyncNode) { await asyncNode.ProcessAsync(context); } else { node.Process(context); }

Context Variable System

Problem

Graphs often need runtime-provided data (e.g., Transform, DeltaTime) that shouldn't be editable in the editor.

Solution: Context Variables

GraphInterop distinguishes between:

  • Parameters (editable in editor, serialized)

  • Context Variables (runtime-provided, read-only)

public class GameplayGraph : GraphAsset { public override IEnumerable<ContextVariableDefinition> GetContextVariables() { yield return new ContextVariableDefinition( name: "transform", type: typeof(Transform), displayName: "Transform", isRequired: true ); yield return new ContextVariableDefinition( name: "deltaTime", type: typeof(float), displayName: "Delta Time", isRequired: true ); yield return new ContextVariableDefinition( name: "input", type: typeof(Vector2), displayName: "Input Vector", isRequired: false // Optional ); } }

Runtime Usage:

var context = new GraphContext(); context.SetContextVariable("transform", transform); context.SetContextVariable("deltaTime", Time.deltaTime); context.SetContextVariable("input", new Vector2( Input.GetAxis("Horizontal"), Input.GetAxis("Vertical") )); await executor.ExecuteAsync(context);

Backend Abstraction

Problem

Tightly coupling to GraphView prevents future migration to GraphToolkit UI.

Solution: IGraphEditorBackend

GraphInterop abstracts the editor implementation:

public interface IGraphEditorBackend { GraphAsset CurrentGraph { get; } void Initialize(EditorWindow window, VisualElement root); void LoadGraph(GraphAsset graph); void OnDisable(); }

Multiple Backend Support:

// GraphView backend (current) public class GraphViewEditorBackend : IGraphEditorBackend { } // Future GraphToolkit backend public class GraphToolkitEditorBackend : IGraphEditorBackend { } // Custom backends public class MyCustomBackend : IGraphEditorBackend { }

Registration:

[InitializeOnLoadMethod] private static void RegisterBackends() { // Priority order: first registered = first tried GraphInteropEditorWindow.RegisterBackend(() => new GraphViewEditorBackend()); GraphInteropEditorWindow.RegisterBackend(() => new GraphToolkitEditorBackend()); }

Benefits:

  • Future-proof architecture

  • Easy to switch backends

  • Custom editor support

  • No user code changes needed

Example: Custom Backend

public class MyDebugBackend : IGraphEditorBackend { public GraphAsset CurrentGraph { get; private set; } public void Initialize(EditorWindow window, VisualElement root) { var label = new Label("Debug Backend Active"); root.Add(label); var nodeList = new ListView(); root.Add(nodeList); } public void LoadGraph(GraphAsset graph) { CurrentGraph = graph; // Update UI } public void OnDisable() { // Cleanup } } // Register [InitializeOnLoadMethod] private static void Register() { GraphInteropEditorWindow.RegisterBackend(() => new MyDebugBackend()); }
