GraphToolkit Features in GraphInterop

This page explains how GraphInterop adapts Unity's GraphToolkit API and which features are supported.

Overview

GraphToolkit (GTK) is Unity's experimental next-generation graph framework. GraphInterop adopts its runtime API design to provide a stable, well-designed foundation for node-based tools.

Why Adapt GraphToolkit?

  1. Official Unity Direction: GTK represents Unity's intended future for graph tools

  2. Well-Designed API: Builder patterns, clear separation of concerns

  3. Runtime + Editor: Designed for both execution and visual editing

  4. Type Safety: Strong typing throughout the API

What GraphInterop Changes

GraphInterop adapts (not wraps) the GTK API:

  • Same API patterns and conventions

  • Stable implementation (GTK is still experimental)

  • Additional features where GTK is incomplete

  • Semantic versioning guarantee

Runtime Core Features

Graph Asset Model

GraphToolkit Original:

// GTK's abstract Graph class [Graph(".mygraph")] public class MyGraph : Graph { public override void OnGraphChanged(GraphLogger logger) { // Validation logic } }

GraphInterop Adaptation:

// GraphInterop's GraphAsset (concrete, not abstract) [CreateAssetMenu(menuName = "My Game/My Graph")] public class MyGraph : GraphAsset { public override IEnumerable<ContextVariableDefinition> GetContextVariables() { yield return new ContextVariableDefinition( "player", typeof(Transform), "Player Transform", isRequired: true ); } }

Key Differences:

  • GraphInterop uses GraphAsset instead of Graph

  • Concrete class, not abstract (can be used directly)

  • ScriptableObject-based for Unity serialization

  • Context variables instead of GTK's validation-only approach

When to Use Options:

Options are for topology-affecting configuration:

// ‚úÖ Good: Affects node structure context.AddOption<int>("outputCount") .WithDefaultValue(3) .Build(); protected override void OnDefinePorts(IPortDefinitionContext context) { int count = this.GetOptionValue<int>("outputCount"); for (int i = 0; i < count; i++) { context.AddOutputPort<float>($"output{i}").Build(); } } // ‚ùå Bad: Should be an input port instead context.AddOption<float>("speed"); // Runtime data, not configuration!

Variable/Blackboard System

GraphToolkit API:

// GTK's IVariable interface public interface IVariable { string name { get; } Type dataType { get; } VariableKind variableKind { get; } // Local, Input, Output bool TryGetDefaultValue<T>(out T value); }

// Create parameter var parameter = graphAsset.Blackboard.AddParameter<float>("Speed"); parameter.Category = "Movement"; parameter.SortOrder = 1; parameter.SetValue(5.0f); // In graph execution var speedParam = graph.Blackboard.GetParameter("Speed"); float speed = speedParam.GetValue<float>(); // Context variables (runtime-provided) public override IEnumerable<ContextVariableDefinition> GetContextVariables() { yield return new ContextVariableDefinition( "deltaTime", typeof(float), "Delta Time", isRequired: true ); }

Execution Features

Graph Execution (GraphInterop Extension)

GraphToolkit: No runtime execution (editor-only in 0.4.0-exp.2)

GraphInterop:

using QhunGaming.GraphInterop; public class MyBehaviour : MonoBehaviour { public GraphAsset graphAsset; private async void Start() { // Create execution context var context = new GraphContext(); context.SetContextVariable("transform", transform); context.SetContextVariable("deltaTime", Time.deltaTime); // Execute graph var executor = new GraphExecutor(graphAsset); await executor.ExecuteAsync(context); // Retrieve results // (nodes set output values in context) } }

Execution Flow:

Graph Execution Steps

  1. Topological Sort: Order nodes by dependencies

  2. Input Resolution: Fetch connected values or use defaults

  3. Node Processing: Execute each node in order

  4. Output Caching: Store results in context for downstream nodes

  5. Completion: All nodes processed

See Graph Executor for details.

Async Node Support

GraphToolkit: Not applicable (no execution)

GraphInterop:

public class LoadAssetNode : GraphNode, IAsyncNode { protected override void OnDefinePorts(IPortDefinitionContext context) { context.AddInputPort<string>("address").Build(); context.AddOutputPort<GameObject>("asset").Build(); } public async Task ProcessAsync(GraphContext context) { var address = this.GetInputValue<string>("address", context); var handle = Addressables.LoadAssetAsync<GameObject>(address); await handle.Task; this.SetOutputValue("asset", handle.Result, context); } }

Supported Async Patterns:

  • async/await

  • Task<T> return values

  • Addressables integration

  • Custom async operations

Best Practices

1. Use Generic Port Builders

// ‚úÖ Good: Type-safe context.AddInputPort<Transform>("target").Build(); // ‚ùå Avoid: Requires manual type handling context.AddInputPort("target") .WithDataType(typeof(Transform)) .Build();

2. Provide Default Values

// ‚úÖ Good: Fallback if no connection context.AddInputPort<float>("multiplier") .WithDefaultValue(1.0f) .Build(); // ‚ùå Risky: May throw if unconnected context.AddInputPort<float>("multiplier").Build();

3. Mark Required Inputs

// ‚úÖ Good: Validates at graph validation time context.AddInputPort<Transform>("target") .Required() .Build();

4. Use Options for Topology

// ‚úÖ Good: Option affects structure context.AddOption<int>("outputCount").WithDefaultValue(3).Build(); // ‚ùå Bad: Should be input port context.AddOption<float>("speed").Build();
